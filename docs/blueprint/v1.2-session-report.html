The arc, in one paragraph

The code was “done.” Everything that broke broke at the seams between environments — laptop vs. Linux, x86 vs. arm64, one bucket vs. another, a file that existed locally but not in git. None of it was logic bugs; almost all of it was packaging, configuration, and contracts between components. That is the real lesson.

The headline bug: a two-day silent AIS outage

How it hid post-mortem

Switching the stack to arm64 fixed the build — but only the predict function got platform-targeted wheels. The other six used a plain pip install that builds for the host architecture. So the compiled pydantic_core shipped as macOS (or x86_64-CI) binaries onto arm64 Lambdas and failed to import. Functions without pydantic (predict, features, export) kept working — which is exactly why the failure was invisible: forecasts still updated, but the three ingestion Lambdas were dead, so no fresh AIS landed. The export’s correct 6-hour vessel window then returned nothing, and the dashboard showed zeros that looked like a quiet port. A second, older bug hid behind it: the un_locode.csv seed had been deleted in a refactor but is still read at import. Fixing one revealed the other.

Lessons — from a junior-dev lens

What this session would teach someone shipping their first production system.

1

Native dependencies are the #1 cloud-deploy trap.

Anything with a compiled core — pydantic_core, lightgbm, scipy, numpy — must match the Lambda’s OS and CPU. Build for the target (--platform manylinux2014_aarch64), never the host.

2

Don’t build production artifacts on your laptop.

A macOS deploy.sh shipping host-arch wheels caused the outage. Either build for the target explicitly, or deploy only from CI — and even CI must target the right arch.

3

Secrets never live in committed config.

One git add nearly leaked the AIS key. Keep it in .env, inject at deploy time, and assert no secret is in tracked files.

4

CI/CD path filters must match what you bundle.

The predict Lambda bundles models/, but the deploy trigger ignored it — so a fixed model contract never deployed and production silently served stale results.

5

“Dead code” may have living dependants.

A refactor deleted the UN/LOCODE seed; ingestion still imported it. Before deleting, grep for importers — and add a test so the contract can’t silently break again.

6

Make silent failures loud.

The dashboard showed zeros with no hint the pipeline was down. A freshness banner turns an invisible outage into an obvious one — observability is a feature.

7

Honesty is a design constraint.

Hardcoded “High congestion”, fake gauges, and heuristic numbers dressed as model output erode trust. Bind every visual to real data or label it as estimated.

8

Verify end-to-end after every deploy.

Each fix was confirmed by an actual invoke / curl / browser check — not assumed. That’s how the null-predictions and the outage were caught at all.
