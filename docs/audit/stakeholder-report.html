TL;DR

The serverless pilot is live and working — AIS, weather, NOAA tides, and macro feeds land in S3, Athena queries them, and a CloudFront-served JSON powers the React dashboard. However, no ML code exists yet (features and models folders are empty), the learning guides target a stack we no longer use (Postgres/Feast/FastAPI), and the proposed refactor plan over-engineers the next step by adding DynamoDB, QuickSight, and a second runtime before the model loop is closed.

Recommendation: reframe the next 4 weeks around closing the ML loop on what already works: Athena features → LightGBM model → Lambda inference → dashboard panel. Defer DynamoDB and QuickSight until a paying use-case demands them.

Where we are today

What ships and works

  • 5 Lambdas orchestrated by EventBridge: AIS (hourly), weather, NOAA tides, daily macro export, freshness watchdog.
  • S3 + Glue + Athena with partition projection (good practice — pay-per-query, zero idle).
  • CloudFront-served demo-data.js consumed by the React dashboard. ~2–5 ms reads, $2/mo CDN.
  • 4 CloudWatch alarms + SNS email; CloudWatch dashboard; S3 lifecycle (Glacier @90d, expire @365d).
  • dbt staging + marts: port_congestion_daily, vessel_daily, 8 dbt tests.
  • React/Vite dashboard v2: 6 pages, MapLibre, Chart.js, Tailwind.

What is missing for the ML promise

  • features/ — empty. No Athena feature views yet.
  • models/training, evaluation, monitoring — empty stubs.
  • No model registry (no MLflow), no drift report (Evidently not wired), no retraining flow.
  • Predictive Analysis dashboard page exists but shows placeholder data.
  • Two parallel runtimes (serverless+Athena vs Prefect+Postgres+FastAPI). The Postgres path is dead code.
  • Target-definition bug in the week-05 guide: a continuous "hours-to-arrival" value is treated as a binary congestion label.

4-week roadmap (weeks 5 → 8)

5

Week 5 — Athena features & targets Rewrite guides

Build Athena CTAS marts: feature_port_status_hourly, feature_vessel_inbound_hourly (10/50/200 nm rings), target_port_congestion_hourly (binary, 75th-percentile threshold per port, 24 h lookahead). One Parquet training-set artifact in S3. Fix the target-definition bug. Drop Feast to an optional advanced track.

6

Week 6 — Train, register, explain

LightGBM binary classifier. Time-based split (last 7 days = test). MLflow tracking with S3 artifact store and SQLite backend. SHAP global + per-prediction. Baseline = "most-recent congestion_score > threshold." Beat it or stop.

7

Week 7 — Serve on Lambda (not Fargate)

Containerize FastAPI as a Lambda container image; Function URL behind CloudFront. Model loaded from S3, cached in /tmp. Predictions written to predictions/date=…/port=…/ and merged into the dashboard JSON. GitHub OIDC for deploys. Evidently drift report generated weekly → S3 → new dashboard tab.

8

Week 8 — Agent & capstone

LangGraph + GDELT news enrichment writes a disruption_score into the same feature mart. Capstone is the existing React dashboard, now backed by real predictions + drift panel + SHAP explanations. No new Streamlit, no new HTML dashboard.

Pre-flight cleanup PRs before week 5 work begins: archive Postgres/FastAPI under legacy/; delete local_copy.ndjson and root-level scratch scripts; remove terraform.tfstate from git; update README status; fix prefect.yaml hardcoded paths.
