2. The six gaps

Gap 1 · Critical  The AI forecast isn't live

In plain terms: the "24h Forecast" card on the dashboard reads a file called predictions.json. That file is only created when someone runs scripts/refresh_predictions.py on their own computer. Nothing in the cloud produces it.

Why it matters: the file in production is dated 2026-05-12 — about a month old — and one port (AEDXB) is blank. So the headline "AI" feature is showing stale numbers that no longer reflect reality.

Evidence: dashboard-v2/src/hooks/usePredictions.js fetches a static /predictions.json; that file is written only by the local script, never by a Lambda.

Gap 2 · Critical  The feature tables don't refresh themselves

In plain terms: before the model can score a port, the raw data has to be rolled up into "feature tables" (vessel ring-counts joined with weather). Those tables are built by running three SQL files by hand. There is no scheduled job that rebuilds them.

Why it matters: even if we made the forecast run in the cloud (Gap 1), it would read a frozen table. The SQL also uses CREATE TABLE IF NOT EXISTS, which means re-running it does nothing once the table exists — it must be dropped first. This is exactly the "table update twice daily" you flagged.

Evidence: athena/queries/*.sql are run manually; no Lambda or schedule references them.

Gap 3 · Critical (silent)  The feature SQL doesn't match what the model expects

In plain terms: the model was trained to expect eight columns with specific names. The checked-in feature SQL produces different names, and is missing one column entirely.

Model expectsSQL actually producesProblem
avg_speed_50nmavg_sog_50nmDifferent name
max_wave_height_mavg_wave_heightDifferent name and meaning (max vs avg)
vessels_at_anchormissingNot produced at all

Why it matters: run the forecast against this table and it errors out ("column not found"). It means the checked-in SQL is not the real source of truth — the table that produced today's predictions had a different shape. We must reconcile them into one contract before automating, or we'll automate a broken pipeline.

Evidence: compare models/features.py + predict.py's query against athena/queries/feature_port_status_hourly.sql.

Gap 4 · Important  The build can break the same way it did before

In plain terms: the Lambda packaging step installs libraries with a plain pip install on your Mac. One of those libraries (pydantic) ships a compiled binary that must be the Linux version — the same mismatch that caused the silent 3-week outage. Today's Makefile has no guard against it.

Why it matters: this is your "containerize everything so data is the same everywhere" goal. The proper fix is to build inside Docker (sam build --use-container) so the package is always built on the real Lambda OS.

Evidence: Makefile uses pip install -r requirements.txt -t with no --platform and no container.

Gap 5 · Cleanup  The old API points at a database we deleted

In plain terms: there's a FastAPI service in serving/ that still tries to connect to a Postgres database. That database was removed when we moved to Athena. The code is dead — it can't run.

Why it matters: it's confusing and it's a trap. Either delete it or rewrite it to read from Athena. Until then it's misleading dead weight.

Evidence: serving/app/core/db.py opens a psycopg pool to POSTGRES_HOST; no Postgres exists anymore.

Gap 6 · Blind spot  Monitoring can't see the forecast step

In plain terms: the dashboards and email alarms watch data coming in (Gap-free) and the export going out. Nothing watches whether the forecast was produced, whether it's fresh, or whether the feature-table rebuild succeeded.

Why it matters: this is your "monitoring of all runs from ingestion to serving" goal. Right now the forecast could silently freeze (it already has) and no alarm fires.

Evidence: template.yaml alarms cover AIS/export freshness and Lambda errors only — no prediction-freshness or feature-build metric.

3. The roadmap — step by step

Five phases, ordered so the highest-value, lowest-risk work comes first. Each phase has a plain goal, the concrete steps, and a clear "done when…" so we know when to stop.

How we'll work (unchanged from your rules): you write the change first, I review and correct it; I never run git — I hand you the commands; we reason through each concept together. This page is the plan only — nothing below has been executed.

Phase 2A — Make the forecast live and correct

Fixes Gaps 1, 2, 3

Highest value · do this first · ~1–2 focused sessions

Goal: the 24h forecast updates itself in the cloud, from fresh tables, using the exact columns the model was trained on.

  1. Reconcile the feature contract (Gap 3 first — it blocks everything). Line up models/features.py against the three SQL files and fix the column names so the table the model reads (feature_port_status_hourly) emits avg_speed_50nm, max_wave_height_m, and vessels_at_anchor. One contract, no drift.
  2. Make the CTAS rebuildable. Switch the three queries from CREATE TABLE IF NOT EXISTS to a drop-then-create (or INSERT into a partitioned table) so a re-run actually refreshes the data instead of silently doing nothing.
  3. Wrap the forecast in a Lambda. Move the logic of scripts/refresh_predictions.py into a new predict_lambda that loads the model from S3 once, scores all 10 ports, and writes predictions.json straight to the public bucket + invalidates CloudFront — the same way export_lambda already does.
  4. Point the dashboard at the cloud file. Confirm usePredictions.js reads the published predictions.json URL, not a file baked into the build.
Done when: you delete the local predictions.json, wait an hour, and the dashboard's 24h forecast still shows fresh numbers with today's timestamp — no laptop involved.

Phase 2B — Refresh the tables automatically, twice daily

Your cost-efficiency goal

Do right after 2A · ~1 session

Goal: feature tables rebuild on a schedule that keeps the forecast fresh without burning money. Twice a day is the sweet spot — the model predicts 24h ahead, so hourly rebuilds add Athena scan cost for almost no accuracy gain.

  1. Add a feature-build Lambda that runs the (now rebuildable) CTAS queries in order: feature_vessel_inbound_hourly → feature_port_status_hourly → congestion_target.
  2. Schedule it twice daily in template.yaml, e.g. cron(0 5,17 * * ? *) (05:00 & 17:00 UTC), and have the predict_lambda from 2A run a few minutes after each rebuild.
  3. Keep scans cheap. The queries already filter to a 30-day window and write Parquet — keep that. Confirm Athena only scans the partitions it needs.
Done when: the curated feature tables show two new partitions per day and the forecast timestamp moves twice a day on its own. Athena cost stays in cents/day.

Phase 2C — Containerize the build (kill the platform bug forever)

Fixes Gap 4 · your Docker goal

~half a session · low risk, high peace-of-mind

Goal: every Lambda is packaged on the real Lambda operating system, so the "works on my Mac, silently dead in the cloud" bug can never come back.

  1. Build in Docker. Switch the deploy to sam build --use-container, which compiles dependencies inside an Amazon Linux image — no Mac/Linux mismatch is possible.
  2. Pin it in the Makefile / CI so nobody can accidentally do a bare-metal build again, and update the deploy docs.
  3. Smoke-test once by deploying and confirming each Lambda imports pydantic without crashing (this is what bit us before).
Done when: a clean sam build --use-container && sam deploy produces working Lambdas, and the README's deploy step says "Docker required."

Phase 2D — Watch the whole pipeline, end to end

Fixes Gap 6 · your monitoring goal

~half a session

Goal: if the forecast or the feature rebuild fails or goes stale, you get an email — the same safety net the ingestion side already has.

  1. Emit metrics from the new Lambdas (2A & 2B): a "prediction freshness" metric and a "feature rows written" metric, mirroring the existing records_written pattern.
  2. Add two alarms: prediction older than ~14h, and feature-build wrote zero rows — both wired to the existing SNS email topic.
  3. Add two widgets to the CloudWatch dashboard so the forecast step is visible next to ingestion.
Done when: deliberately skipping a forecast run triggers an email within the alarm window, and the dashboard shows the forecast freshness line.

Phase 2E — Clear out the dead serving code

Fixes Gap 5 · housekeeping

~quick · do last or whenever

Goal: no misleading code that points at infrastructure we deleted.

  1. Decide: do we still want a live HTTP API, or is the published JSON enough? (Recommendation: the static JSON already serves the dashboard — we likely don't need the API yet.)
  2. If not needed: remove serving/app/ and its Postgres settings.
  3. If needed later: rewrite db.py/queries.py to read from Athena instead of Postgres, and drop it behind the same auth as the dashboard.
Done when: there is no code referencing POSTGRES_* anywhere in the repo.

  2. The old API: delete serving/ now, or keep it parked for a future Athena rewrite?
