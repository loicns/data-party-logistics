# Dashboard Publishing Architecture

This page explains how the dashboard UI and the live export artifact fit together.

## Current Split

The project currently has two different dashboard concerns:

### Static UI assets

- `dashboard/index.html`
- `dashboard/app.js`
- `dashboard/styles.css`

These are the frontend shell.

### Live data artifact

- `exports/dashboard/demo-data.js`

This is generated by the export Lambda and contains `window.DEMO_DATA`.

## Why The Split Exists

The pilot deliberately avoided building:

- a live API
- a long-running backend server

Instead, it uses a **static artifact serving model**:

- Athena queries raw data
- export Lambda builds a payload
- dashboard reads that payload as JavaScript

That is cheaper and simpler than standing up a full API for the pilot.

## Current Publication Path

Current state:

- the export Lambda writes `demo-data.js` into the pilot data bucket
- the UI shell can be hosted separately as static files

This means the live dashboard requires:

1. UI assets to be hosted somewhere
2. `demo-data.js` to be reachable by that UI

## Recommended Low-Cost Hosting Shape

### UI bucket

Use a separate static website bucket for:

- `index.html`
- `app.js`
- `styles.css`

### Data artifact

The cleanest next improvement is:

- have the export Lambda also publish `demo-data.js` into the public dashboard bucket

That avoids manual copy steps and keeps the dashboard synchronized with scheduled exports.

## Short-Term Versus Desired State

### Short-term acceptable

- manually sync dashboard assets to S3
- manually or separately copy `demo-data.js`

### Desired pilot state

- GitHub `main` deploys dashboard assets automatically
- export Lambda publishes fresh `demo-data.js` automatically
- dashboard refresh requires no manual artifact movement

## What This Is Not Yet

- a CDN-optimized production web app
- a full API-backed dashboard
- an authenticated product surface

It is a cheap, honest publication path for a pilot artifact.
