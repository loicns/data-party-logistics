Detailed Task Board

Open this only when you need implementation detail. The board stays collapsed by default to keep the high-level roadmap clean.

Task Tracks
Track A: AWS foundation Make deployment safe and low-cost before the first run.
Create deploy role

GitHub OIDC trust, CloudFormation deploy permission, Lambda/S3/Glue/Athena/CloudWatch/SNS permissions.

Maps to M2
Use OIDC instead of static AWS keys so GitHub deploys stay safer and cheaper to operate.

Confirm least-privilege scope, trust policy to GitHub, and explicit access to the exact services the SAM stack creates. This is the main identity control point for the pilot.

Review region defaults

Keep stack and workflow anchored to eu-west-3 to match cost assumptions.

Maps to M2
Avoid accidental region drift between GitHub workflow, manual console checks, and stack outputs.

A region mismatch breaks price assumptions and can create “why is nothing updating?” confusion if you inspect the wrong AWS console region later.

Guard against cost leaks

Do not add NAT, RDS, Fargate, or VPC-only complexity to this pilot path.

Maps to M2
This is the most important architecture discipline for the whole pilot.

If somebody proposes RDS, Fargate, or a VPC plus NAT for convenience, the correct response is to pause and revalidate the mission, because that changes the operating model and cost structure.

Track B: Secrets and deploy Make the browser-first deployment path actually usable.
Add GitHub secrets

AWS_DEPLOY_ROLE_ARN, AISSTREAM_API_KEY, optional DPL_ALERT_EMAIL.

Maps to M3
These secrets are the minimum required inputs for the first live deployment.

Prefer GitHub environment scoping if you expect multiple stages later. For this pilot, repository-level secrets are enough as long as the deploy role is tightly scoped.

Run workflow manually

Use the deploy workflow once before trusting push-based automatic updates.

Maps to M3
The first deployment should be deliberate and observable, not hidden inside normal code pushes.

Watch workflow logs, CloudFormation events, and stack outputs at the same time. This is where you catch trust, permission, or schedule-definition issues fast.

Confirm stack outputs

Bucket name, Athena workgroup, dashboard name, and export artifact path.

Maps to M3
If the outputs are ambiguous, later monitoring and artifact verification will be slower and more error-prone.

Write down the exact output names and URLs. This becomes the operator map for the first live validation cycle.

Track C: Data runtime Verify ingestion, querying, and export with real landed files.
AIS hourly snapshot check

Confirm Lambda duration, S3 file creation, record counts, and no silent zero-output runs.

Maps to M4
This is the most important runtime proof because AIS is the highest-frequency and most timing-sensitive source.

The Lambda should stay inside the 5 to 7 minute window, write at least one object, and avoid the dangerous case where the run succeeds technically but produces no usable data.

Weather and NOAA landing

Check raw prefixes, data shapes, and Glue/Athena compatibility with actual JSON objects.

Maps to M4
Side-source quality matters because export logic and alerts depend on fresh signals, not AIS alone.

Validate the actual JSON shape against Glue table definitions and check that partitions or prefixes remain predictable enough for low-cost Athena querying.

Export validation

Confirm Athena-backed export writes both demo-data.js and curated JSON output.

Maps to M4
This is the bridge between the data pipeline and the user-visible dashboard surface.

Watch Athena scan size and query time here. If costs start drifting, this is the most likely optimization point before touching the ingestion architecture.

Track D: Monitoring and product Turn cloud events into a usable operating surface.
Confirm CloudWatch metrics

AIS files written, freshness minutes, export writes, and Lambda errors appear as expected.

Maps to M5
If the dashboard is empty or ambiguous, you lose the whole “useful-at-a-glance” operating advantage.

Focus on a small metric set first. The goal is actionable operator feedback, not an encyclopedic monitoring setup.

Test alarms

Verify missing AIS, stale export, and Lambda failure paths can notify through SNS.

Maps to M5
Alarm paths are part of product quality here because you will trust the pilot less if silent failure is possible.

At least once, simulate a stale path or failed invocation so you know the alerting story works before users depend on the system.

Choose dashboard hosting

Decide whether the static artifact lives behind S3 website hosting, CloudFront, or another cheap path.

Maps to M5
The cheaper answer is usually enough for the pilot as long as the artifact is refreshed reliably.

Prefer the simplest hosting path that gives you refresh reliability and acceptable access control. Over-optimizing hosting too early is usually wasted effort for a three-port pilot.

Danger Zone, Guardrails, And Definition Of Done

This section is styled to be harder to ignore on purpose. These are the moves most likely to break either the cost model or the operational clarity of the pilot.

Warning

What can break the low-cost promise

  • Adding RDS back into the pilot runtime
  • Adding Fargate or long-lived workers
  • Adding NAT gateways
  • Letting Athena scan unbounded raw JSON
  • Very noisy CloudWatch logs or long i TTL values on data you do not need for long

Checks before calling it live

  • One full hourly cycle succeeds end-to-end
  • AIS writes at least one file each run
  • Export freshness stays within threshold
  • Dashboard artifact refreshes from S3
  • Alarms can notify through SNS

Definition of done for pilot go-live

The pilot is live when the stack is deployed, GitHub-triggered deploys work, the CloudWatch dashboard shows real data, the export artifact updates on schedule, and measured cost after a few days still fits inside the intended low-cost envelope.
