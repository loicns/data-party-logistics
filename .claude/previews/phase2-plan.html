Serving-layer reality check

Before you touch serving/app/, know what you're walking into. I read it — it's built for infrastructure you already deleted.

The trap: serving/app/core/config.py requires postgres_host, postgres_user, etc. Your routes.py runs SQL via get_conn() against Postgres. But you deleted the RDS instance and migrated to Athena (commits e469b7c, 60d2f78). This FastAPI app cannot run as-is — it connects to a database that no longer exists.

What that means for "Option B"

Assumption in the guideReality

"Quick endpoint"You'd rewrite core/db.py + core/queries.py from Postgres to Athena, then add a new model-backed /forecast route.

The honest path when you reach 2D

1
Decide: revive FastAPI, or go serverless
Option B-lite: skip the FastAPI app entirely. A single Lambda + API Gateway that calls your predict() function against Athena. Far less code than reviving the Postgres app.
2
Add a model-backed forecast endpoint
GET /forecast/{port} → loads model_latest.pkl, queries latest Athena row, returns the prediction dict. This is predict.py behind HTTP.
3
Frontend fetches the live endpoint
Point usePredictions() at the API Gateway URL. Now predictions are true real-time, per request.
Architect's call: for 10 ports refreshed hourly, 2B (scheduled S3 file) beats 2D (live API) on cost and simplicity. A real-time API only earns its complexity if you need sub-hourly, on-demand predictions. You probably don't yet. Build 2B; treat 2D as a "when there's real demand" upgrade.
