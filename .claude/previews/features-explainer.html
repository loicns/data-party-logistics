Why port_code and observation_hour are not features

And what you actually do with them — plain English with examples.

The core rule

A model is math. It can only learn from numbers.

What LightGBM does internally (simplified)
prediction = vessels_in_10nm × 0.31 + vessels_at_anchor × 0.44 + hour_of_day × 0.12 + …
prediction = "BEANR" × ???   ← can't multiply a string
prediction = 2026-05-11 18:00:00 × ???   ← multiplying a timestamp is meaningless

The model finds patterns by doing arithmetic on columns. You can't multiply a port name or a full timestamp by a weight — there's no sensible number to put there.

But what DO you use them for?

Each column has a job — just not "feature."

❌ port_code as a raw feature

String "BEANR" → model can't learn from it. Different ports have totally different congestion patterns, but the model can't see that from a text label.

✅ port_code → one-hot encoded

Turn it into numbers: is_BEANR=1, is_NLRTM=0, is_SGSIN=0 …. Now the model can learn "BEANR congests differently than SGSIN." That's v2 — not needed for MVP.

❌ observation_hour as a raw feature

A full timestamp like 2026-05-11 18:00:00 is meaningless to the model. It can't learn "18:00 is rush hour" from a raw datetime — it'd just see a big integer.

✅ observation_hour → already extracted!

You already did this in the SQL: hour_of_day=18, day_of_week=7. The model learns from those — the timestamp's job is done. It only stays in the df for the sort step.

The split in code is one line:
Sort df by observation_hour ascending. Then: first 80% of rows → train, last 20% → test. No shuffle. No randomness. After the split, observation_hour and port_code are dropped from X — they were never in FEATURES to begin with.
Full picture — every column's role

Where each column actually goes

vessels_at_anchor→ FEATURE (strongest signal)
